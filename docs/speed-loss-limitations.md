# Speed Loss 已知限制與誠實聲明

本文件說明 `ml/speed_loss.py`(ISO 19030 in-service speed loss + 船殼/螺旋槳歸因)在**這批黑客松資料**上的限制。這些限制是資料本質造成的,並非實作錯誤;主動揭露反而展現方法的嚴謹。

## 1. 汙損 → 油耗的訊號本質偏弱

以資料驗證:同船、相同 RPM 分箱內,`foc_eq24` 與 `days_since_hull` 的相關係數僅約 **+0.12**;油耗模型的 permutation importance 中,養護時鐘(`days_since_hull/prop/dd`)與汙損旗標皆 **< 0.005**(RPM 為 0.94,一家獨大)。

**影響**:真實汙損效應大約落在 1–5% 量級,而日次 speed loss 的雜訊約 ±5%。因此:
- 「事件前後 30 天窗口的 recovery 差」訊噪比低,不足以當主證據 → 我們改用**汙損累積率(interval 內斜率)**與**結構性時鐘證據**。
- speed loss 曲線需經移動平均(`roll_speed_loss_pct`)才穩定,單日值僅供參考。

## 2. 負的 speed loss 代表「無可測退化」,不是變乾淨

部分清洗久遠的船(如 S1、S3、S7)最新 speed loss 呈輕微負值(約 −2% ~ −5%)。

**解讀**:相對「該乾塢間隔清洗後基準」,近期在雜訊範圍內**無可測退化**;負值多來自運轉型態變化(例如後期較常低速航行,經 Admiralty `log_v3d23` 正規化後仍有殘餘)。**不應解讀為船體變乾淨**。前端建議把 −2% ~ +2% 視為「持平/無顯著退化」帶。

## 3. 部分船隻歸因為 INDETERMINATE

hull/propeller 歸因採 ISO §4.3 要求的 **thrust 法**。當近端窗口內 thrust/RPM 有效樣本不足,或殘差皆非正(無可測退化)時,回報 `INDETERMINATE`(而非硬湊一個百分比)。

**影響**:少數船(視資料期間而定)的 `hull_contribution_pct` / `propeller_contribution_pct` 為 `null`。這是誠實標示,前端應顯示為「歸因資料不足」而非 0%。

## 4. 功率以燃油代理(非扭矩量測)

ISO 19030-2 §4.2 首選以軸扭矩計算交付功率;本資料無可靠扭矩/SFOC(大量缺失),故依標準允許的 **brake-power 路徑**,以 `foc_eq24`(VLSFO 當量油耗/24h)作為功率代理。

**影響**:speed loss 由「功率增幅 ÷ 3」導出(P ∝ V³),屬功率等價的速度損失,而非直接量測的對水速度損失。量級與趨勢可靠,絕對值有此換算假設。

## 5. 參考期依賴養護事件

每艘船的「乾淨參考曲線」以**船殼重置事件(DD/UWC/UWC+PP)後 45 天穩態日**擬合。若某船缺早期清洗事件,退回「`days_since_hull` 最小 25%」或「最早 30% 穩態日」為近似乾淨基準。

**影響**:退回近似基準的船,其 speed loss 絕對水準的可信度較低(趨勢仍有效)。`ships[].reference.n_ref_days` 揭露每船參考樣本數,可作信心指標。

## 6. UWI 陷阱:為何我們用結構性證據而非 recovery 數字

命題方在資料中埋入純 **UWI(僅水下檢查,無清洗)** 事件,誘導參賽者在檢查後畫出效能改善。

我們**不**依賴脆弱的「事件前後 recovery」來證明沒中招(見第 1 點,該指標訊噪比低)。而是用**結構性、100% 確定**的證據:每個事件標記 `days_since_hull_at_event` ——
- 真正的船殼清洗:當天 = **0**(維護時鐘歸零)
- 純 UWI:當天 = **一大串天數**(中位數約 668 天,時鐘不歸零)

因此我方 speed loss 曲線在純 UWI 後**不會**出現虛假的階梯下降。這是不可辯駁的邏輯證據,獨立於資料雜訊。

---

**摘要**:speed loss 的**趨勢、跨船排序、UWI 判斷、汙損累積率**是穩健且可信的;**單日絕對值、弱訊號船的 recovery 數字**應保守解讀。所有信心指標(`n_ref_days`、`has_thrust_model`、`attribution=INDETERMINATE`、`fouling_rate`)都已寫入 `speed_loss.json`,供前端與評審檢視。
